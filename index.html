Why Container Queries?

The pattern used to build this page â€” and the right tool for any component that needs to respond to where it's placed, not how wide the screen is.

The old approach â€” viewport-dependent

/* Fires when the *screen* is narrow. Breaks if the card sits in a sidebar on a wide monitor â€” query never fires. */ @media (max-width: 600px) { .card-inner { display: block; } }

The container query approach

/* Step 1: register the card as a container */ .device-card { container-type: inline-size; container-name: card; } /* Step 2: query the card's OWN width */ @container card (min-width: 400px) { .card-inner { display: grid; grid-template-columns: 156px 1fr; } }

Card fires at 400px of its own available space. Whether it's in a full-width grid, a two-column layout, or a modal â€” the query works correctly because it measures the card's box, not the viewport.

What you'll see on this page

Narrow your browser window to around 640px. Cards that are smaller than 400px switch from the side-by-side (image left, text right) layout back to a stacked layout. Cards that are still wider than 400px â€” for example if you're in a three-column grid â€” stay horizontal.

That independence between cards is what media queries cannot give you. It's also why the spec was prioritised as a CSS Baseline feature: every major browser ships it, and it's safe to use without fallbacks today.

Key rule for dense data UIs

Use container-type: inline-size on the wrapper, not on the card itself â€” the container cannot query its own size, only its children can. The wrapper becomes the measuring box; children query it with @container.
